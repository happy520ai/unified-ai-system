# 逆熵元法则 — AI Gateway Workbench 第一性原理

> **核心断言：** 系统的唯一底层法则是构建局部的、暂时的、能自我维持的**低熵秩序**。
> **永恒检验：** 此刻的决策是在注入负熵，还是在加速熵增？

本文档是本系统的最高设计原则，不朽且唯一。所有架构决策、代码变更、产品演进均须以此为准绳。

---

## 一、元法则：逆熵

系统的一生，是与熵增永恒抗争的进程。不能逆熵的，即刻朽坏；能持续逆熵的，即为活的系统。

**唯一检验问题：** 我此刻的决策，是在为系统注入负熵，还是在加速熵增？

**终极目标：** 以最低能耗，维持最高秩序。

---

## 二、七大投影（操作铁律）

### 1. 原点律 — 目的即引力源

**本系统的最小目的：** AI Gateway Workbench — 本地动作可执行，模型调用受控。

一切增删以此为唯一试金石。功能蔓延、目的漂移、需求膨胀均为熵增。

**硬约束：**
- 新增模块必须回答：你服务于「本地可执行 + 模型受控」中的哪一条？
- 无法回答的模块，不得创建。
- Phase 命名必须绑定明确业务目的，禁止「占位 Phase」。

### 2. 边界律 — 接口即熵垒

边界是秩序与非秩序的分界线。内部可复杂，外部须极简。

**硬约束：**
- HTTP API 错误响应统一为 `{ status, error: { code, message, details? }, meta }` 格式。
- 鉴权头统一为 `Authorization: Bearer <token>`。
- 环境变量仅允许 `KNOWLEDGE_*`、`AI_GATEWAY_*`、`PGVECTOR_*` 前缀。
- WebSocket 错误、SSE 错误必须复用同一 envelope 结构，禁止发明第四种格式。
- packages/ 不得 import apps/ 的任何文件。
- 跨包引用必须通过包的公开入口（package.json main），禁止深路径穿透。

### 3. 分层律 — 解耦即封装

将宏观失控的复杂性，分解为微观可治理的独立低熵模块。

**硬约束：**
- 五层架构：UI -> HTTP Server -> Business Logic -> Adapters -> Infrastructure。
- 层间靠抽象协议对话，禁止越层调用。
- 单文件不得超过 500 行。超过必须拆分。
- 禁止上帝模块（单文件承担 3+ 不同职责）。
- 禁止循环依赖。

### 4. 闭环律 — 反馈即修复

反馈是系统的感知与自适应机制。

**硬约束：**
- 每条错误输出必须有对应监控。错误不得被静默吞没。
- 已建立四环：Health Check / Chat 错误 / Model Config 检测 / Audit Log。
- 待建立三环：Provider 健康度监控 / Knowledge 检索质量反馈 / Token Usage 预警。
- 每个 `catch` 块必须记录或上报，禁止空 `catch {}`。

### 5. 剃刀律 — 极简即净化

任何非必要的实体，本身就是一块新引入的熵。

**硬约束：**
- 同一函数不得在 2+ 文件中重复定义。必须提取到共享模块。
- 同一功能的函数不得有不同命名（如 `readJsonIfPresent` vs `readJsonIfExists`）。
- 死代码发现后立即清理。
- 过时中间态文档发现后立即归档或删除。
- entrypoints/ 中的验证脚本必须从共享工具导入 `sleep`、`fetchJson`、`readJson`、`writeEvidence` 等通用函数。

### 6. 涌现律 — 自组织即进化

简单规则能自下而上涌现出灵活健壮的宏观秩序。

**硬约束：**
- Provider 适配器模式：新增 Provider 只需实现标准接口，不改核心路由。
- Knowledge 存储插件化：通过配置切换，不改检索接口。
- Workforce Role 组合：角色可组合，不硬编码角色列表。
- 禁止中央调度过强。禁止硬编码不可扩展逻辑。

### 7. 耗散律 — 开放即永生

系统必须持续从外界引入负熵，向外排出高熵废物。

**硬约束：**
- 已建立：日志轮转、Session 过期、Evidence 压缩。
- 待建立：Model Cache LRU 淘汰、Knowledge Chunk GC、Approval Request 过期、Temp File 统一清理。
- 封闭系统必然走向死寂。必须持续吸纳新输入，果断淘汰老旧模块。
- `legacy/` 目录仅作为只读参考，不得新增文件。

## 三、执行约束（每次变更前自检）

1. **目的自检：** 此变更是注入负熵还是加速熵增？
2. **归属自检：** 新增模块归属五层之几？不得越层调用。
3. **边界自检：** 新增路由是否实现标准错误格式？
4. **体量自检：** 新增文件是否超过 500 行？超过须拆分。
5. **重复自检：** 是否复用了已有共享函数？还是又复制了一份？
6. **反馈自检：** 新增的 catch 块是否记录或上报了错误？
7. **代谢自检：** 是否有老旧代码需要清理？
