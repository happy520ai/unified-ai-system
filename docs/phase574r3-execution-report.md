# Phase574R-3 Execution Report

## Summary

Phase574R-3 adds a compatibility guard around the Workbench approval-list
initialization path. The purpose is to prevent
`workbenchApiClient.listApprovals is not a function` from becoming a fatal
Mission Control initialization toast in local dry-run builds.

## Modified Files

- `apps/ai-gateway-service/src/ui/consolePage.js`
  - Added `state.approvalsUnsupported`.
  - Added a `typeof workbenchApiClient.listApprovals === "function"` guard in
    `loadApprovals()`.
  - Added a non-fatal local dry-run unavailable message for the approvals panel.
- `tools/phase574r3/validate-workbench-init-compatibility-guard.mjs`
  - Added real Chromium/CDP validation for no init error, sample dry-run click,
    detail drawer click, safety boundary, and character-module absence.
- `tools/phase574/validate-scenario-based-first-real-trial-experience.mjs`
- `tools/phase574r/validate-actual-clickable-sample-dry-run-repair.mjs`
- `tools/phase574r2/validate-first-screen-sample-entry-ux-lock.mjs`
  - Added the same conservative Chromium software-rendering flags used by
    Phase574R-3 so these existing browser verifiers remain runnable in the
    current Windows headless environment. No assertions were weakened.
- `docs/phase574r3-workbench-init-compatibility-guard.md`
  - Added root cause, fix description, safety boundary, evidence, and rollback.
- `docs/phase574r3-execution-report.md`
  - Added this execution report.
- `apps/ai-gateway-service/evidence/phase574r3/`
  - Generated by the verifier.

## Browser Verifier Environment Note

During regression, the older Phase574 browser verifiers initially failed before
DOM inspection with `Target crashed` or a transient CDP `fetch failed`. Browser
stderr showed repeated GPU process crashes ending in `GPU process isn't usable`.
The verifier launch flags were tightened to no-sandbox, single-process, and
SwiftShader/software rendering. After that, Phase574R-2, Phase574R, Phase574,
and Phase574R-3 all passed their real Chromium checks.
