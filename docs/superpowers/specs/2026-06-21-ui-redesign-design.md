# UI 重构最终方案 v5：对话协议（这是范式终点）

> **日期**：2026-06-21（v5 范式终点，基于真实当前代码 + 截图 + chat-gateway 本质）
> **范围**：`unified-ai-system` 全部前端 UI + 多入口
> **状态**：**范式层面最终方案**。之后不再有 v6 的范式跃迁，剩余提升属"实施中像素调优"。
> **遵循契约**：`PME_DESIGN.md`、`DO_NOT_DESIGN_LIKE_THIS.md`、`COMPONENT_PATTERNS.md`、`phase1181-1200-future-minimal-os-visual-spec.md`

---

## 0. 一句话总结

> 停止设计"首页"。这个系统**就是一个 chat gateway**——让它做回自己：**第一屏就是一次对话的开始**。状态、问候、呼吸灯全部合并进"系统对你说的第一句话"里。你回话，系统用整个 agent workforce 响应。UI 不是一面展示信息的墙，而是一根"你一句我一句"的电话线。

## 1. v4 的盲区与 v5 的纠偏（基于真实证据，非臆测）

### 1.1 v4 用了 800 个 Phase 前的截图

v4 诊断基于 `phase1181_1200`（约 Phase 1181）。但当前代码（`apps/ai-gateway-service/src/ui/components/OwnerOSShell.js`）已迭代到 Phase 1900+。真实当前状态：

- 标题"总控台" + 副标"正在加载系统状态…"
- 主按钮"启动检查"
- **4 个环形指标卡**：服务健康 / 活跃模型 / **Provider** / 安全状态
- 反馈行 + 三张结果卡（今天完成/发现问题/下一步）

### 1.2 ⭐ v4 的"切浅色"可能是在解决已解决的问题

真实截图（`phase1858-owner-os-home.png`）显示**当前 Owner OS Home 已是浅色**。v4 花大篇幅论证"切浅色修漂移"——但当前可能已是浅色，漂移在 Phase1181→1858 之间已被部分修复。**v5 不再把"切浅色"当核心卖点**，而是作为"维持并统一现有浅色基调"。

### 1.3 真实当前代码暴露的术语外泄（v4 说对了，但要落实到具体行）

`OwnerOSShell.js:63` —— 一个**指标卡标签直接写 "Provider"**（英文，给老板看）。这正是 `PME_DESIGN` 禁止的"暴露 Provider 给 owner"。v5 的词汇表硬墙**必须覆盖这类具体 case**，不只是泛泛而谈。

## 2. v5 的核心跃迁：从"首页"到"对话协议"

### 2.1 根本洞察——这个系统就是一个 chat gateway

四版方案我都在把一个 chat-gateway 系统设计成"不是 chat"的东西（仪表盘/任务流/ambient 墙）。但它**就是一个 chat gateway**（`/chat`、`/chat-gateway/execute`、`chatGateway`、WebSocket `/ws`、`chatActionIntentParser`）。

**让它做回自己：第一屏就是 chat，只是这个 chat 背后站着一整个 agent workforce。**

### 2.2 顶级 agent 产品都没有"首页"，只有"第一句话"

| 产品 | 第一屏 |
|---|---|
| ChatGPT | 输入框 + "What can I help with?" |
| Claude | 输入框 + "How can I help you today?" |
| Cursor | 你上次没写完的代码 |
| Linear | 今天的任务列表 |

**它们都不"展示自己"，它们"开始对话"。**

### 2.3 从"墙"到"电话线"

- **首页范式（v1-v4）**：UI 是一面**墙**——展示呼吸灯/问候/指标/卡片/输入框。墙需要设计布局、密度、层次。所以四版都在纠结密度/焦点/折叠。
- **对话协议（v5）**：UI 是一根**电话线**——你一句我一句。电话线没有"密度"问题（对话是线性的），没有"布局"问题（一句话接一句话），没有"术语墙"问题（对话用自然语言）。

**消灭"首页"概念，所有 v1-v4 纠结的密度/焦点/折叠问题自动消解。**

## 3. 对话协议的具体形态

### 3.1 第一屏：只有系统对你说的第一句话 + 你回话的地方

```
╔═══════════════════════════════════════════════════════════╗
║                                                            ║
║                                                            ║
║                                                            ║
║    早上好。今天一切正常，系统随时待命。                       ║  ← 系统的第一句话
║    想做点什么？                                              ║    （状态+问候合并进语气）
║                                                            ║
║    ╭──────────────────────────────────────────╮           ║  ← 你回话的地方
║    │                                            │           ║    （不是"输入框"，是"回话"）
║    │                                            │           ║
║    ╰──────────────────────────────────────────╯           ║
║                                                            ║
║    或试试：看看今天日报 · 审批那 3 项 · 员工在干嘛            ║  ← 建议（融入对话语气）
║                                                            ║
║                                                    ⋯ 更多  ║  ← 安全网（v4 保留）
╚═══════════════════════════════════════════════════════════╝
```

**对比 v4 第一屏的关键变化**：
- 呼吸灯 + 问候 + 输入框 = **3 个元素** → **1 句话 + 回话处**（呼吸灯融入"今天一切正常"，问候融入"早上好"）。
- 不再有"展示"，只有"对话开始"。
- 状态不再是独立元素（呼吸灯），而是**藏在系统说话的语气里**：
  - 正常："今天一切正常，想做点什么？"（轻松）
  - 待审批："有 1 件事等你确认，先看看？"（提醒）
  - 异常："刚才有个任务失败了，要看看怎么回事吗？"（关切）

### 3.2 状态如何"藏在语气里"（消灭呼吸灯这个独立元素）

| 系统状态 | v4 的呈现（独立元素）| **v5 的呈现（融入第一句话）** |
|---|---|---|
| 正常 | 绿呼吸灯 + "今天一切正常" | "早上好。一切正常，想做点什么？" |
| 待审批 | 黄呼吸灯 + 异常卡 | "早上好。有 1 件事等你确认——调 DeepSeek 生成周报，风险低。批准吗？" |
| 异常 | 红呼吸灯 + 异常卡 | "早。刚才有个任务没跑成，我记下来了。要看看吗？" |
| 离线 | 灰呼吸灯 | "系统暂时连不上，我在重试。你可以先告诉我待会儿要做什么。" |

**状态不再是 UI 上的一块"显示区域"，而是对话的上下文。** 这才是真正的 ambient——信息存在，但不占地方。

### 3.3 对话如何驱动 agent workforce（v5 的力量所在）

你说一句话，系统不只是"回答"，而是**调度背后的 workforce**：

```
你:   帮我把昨天客户反馈归类，找前三类问题

系统: 好的。这个我会用"严谨"的方式做（多对比几个方案，避免遗漏）。
       大概 5 分钟。做的时候不用盯着，做完我叫你。

       ▍（背后：product-chief 接单 → 三路分支 → DeepSeek 生成 → ux-researcher 复核）

你:   [关掉浏览器去做别的]                                    ← fire-and-forget（v3）

[5 分钟后，企微推送]
系统: 客户反馈归好了。前三类是：①发货延迟 ②价格异议 ③产品质量。
       要看完整结果吗？

你:   [点推送回到网页]
       ┌─ 客户反馈归类 ──────────────────────────────┐
       │ 完成。用"严谨"方式（3 方案对比，无冲突）。       │
       │                                                │
       │ 前三类：                                       │
       │   ① 发货延迟（47 条）                           │
       │   ② 价格异议（31 条）                           │
       │   ③ 产品质量（22 条）                           │
       │                                                │
       │ [查看证据]  [导出表格]  [继续聊]                  │
       └────────────────────────────────────────────────┘
```

**这是对话，不是表单。** `God Mode` → "严谨的方式"（自然语言）。进度 65% → "大概 5 分钟，做完叫你"（口语）。`Provider`/`CredentialRef` → **永远不会出现在对话里**。

## 4. v5 与 v1-v4 的关系：吸收，不是替换

v5 **不是否定 v1-v4**，而是把它们收束到一个更根本的范式里：

| v1-v4 的设计 | v5 如何收束 |
|---|---|
| v1 角色/IA | → "更多"安全网抽屉里（对话是主路径，IA 是 fallback）|
| v2 任务流/深度旋钮 | → 对话的"展开"就是深度（你说"看看细节"→ 展开证据）|
| v3 输入框+异步+多入口 | → 输入框就是"回话处"，异步就是"做完叫你"，多入口不变 |
| v4 浅色+单焦点+术语墙+气泡 | → **全部保留**（对话天生是单焦点、无术语、气泡形态）|
| v4 词汇表硬墙 | → **强化**：覆盖 `OwnerOSShell.js:63` 的 "Provider" 这类具体 case |

**v5 = v4 的所有像素级修正 + 一个更根本的范式（对话协议）把它们组织起来。**

## 5. 视觉与组件（沿用 v4 浅色，重组为对话形态）

### 5.1 视觉基调（维持 v4，不重复论证）
浅色冷调（`#fafbfc` 背景 / 半透明白表面 / 蓝→紫 CTA），遵 Phase1181 规范。v5 不改 v4 视觉，只重组布局为对话。

### 5.2 v5 新增/重组组件

- `ConversationShell` — **取代** OwnerOSShell 的"首页"布局。第一屏 = 系统第一句话 + 回话处 + 建议 + 更多。
- `SystemFirstLine` — 系统的第一句话（状态+问候合并，语气化，§3.2）。**取代** 独立呼吸灯+问候语。
- `ReplyField` — 回话处（v4 的对话气泡，v5 重新定位为"对话的一部分"而非"输入框"）。
- `ConversationTurn` — 一轮对话的渲染单元（系统说/用户说/系统响应）。
- `WorkforceActionDisclosure` — 对话里轻量披露背后的 workforce 动作（"用严谨方式做""product-chief 接单了"），可点击展开证据。
- `ToneByStatus` — 状态→语气的映射器（§3.2 四态四语气）。

### 5.3 孤立 Panel 整合（v4 映射升级）
- OwnerOSShell → **ConversationShell**（不是"改首页"，是"换范式"）。
- 4 个指标卡（含 "Provider"）→ **删除**，状态融入 SystemFirstLine。`Provider` 永不外露。
- 三张结果卡 → 只在系统"有结果要告诉你"时作为对话的一轮出现，不再常驻。

## 6. 交互铁律（沿用 v1-v4）
1. 审批驱动，非点击即生效。
2. 每个按钮给反馈。
3. 每个操作可追证据。

对话协议天然兼容：审批 = 系统在对话里问你"批准吗？"；反馈 = 对话里的回应；证据 = "查看证据"是对话里的一个动作。

## 7. 实施路径（v5，对话协议优先）

⚠️ **诚实声明**：v5 的核心是**范式重组**（OwnerOSShell → ConversationShell），不是从零建设。后端 chat gateway 已就绪，工作量集中在 UI 范式切换 + 词汇墙 + workforce 对话披露。

| 迭代 | 工作项 | 后端支撑 | 价值 |
|---|---|---|---|
| **P0-a** | ConversationShell（SystemFirstLine + ReplyField + ToneByStatus）| ✅ chat gateway | **范式切换，消灭"首页"** |
| **P0-a** | 词汇表硬墙（覆盖 `OwnerOSShell.js:63` Provider 等）| ✅ | 术语根除 |
| **P0-b** | WorkforceActionDisclosure（对话里轻量披露 agent 动作）+ 建议 chip 动态化 | ✅ workforce | 对话有"力量感" |
| **P0-b** | "更多"安全网抽屉（v4 IA fallback）+ 深度旋钮 | ✅ | 安全网 |
| **P1** | fire-and-forget + 完成推送 + 回放（作为对话的延续）| 队列✅/推送✅ | 异步闭环 |
| **P2** | 多入口（桌面组件/IM/语音/邮件）| 部分需建 | 五入口 |
| **P2** | 浅色空间劳动力画布（"看看员工在干嘛"召唤）| — | 签名概念 |
| **P3** | 路由/知识/安全/清理 | — | 补全 |

## 8. 验收标准（v5 新增）

沿用 v4 全部守卫，**新增**：
- **无首页守卫**：第一屏不存在"展示性元素堆叠"（呼吸灯+问候+指标+卡片+输入框这种墙）。第一屏只有 SystemFirstLine + ReplyField + 建议 + 更多。
- **状态入语气守卫**：系统状态不以独立 UI 元素呈现，而是融入 SystemFirstLine 的语气（自动化测试：无独立 `.breathing-light`/`.status-pill` 元素，状态来自 ToneByStatus 输出的文案）。
- **Provider 字面消失守卫**：DOM 扫描，老板视图无 `Provider`/`God Mode`/`Tianshu`/`CredentialRef`/`sandboxed`/`evidence`/`trace`/`phase`/`verifier` 原词。**具体覆盖 `OwnerOSShell.js:63` 这种 case。**
- **对话力量守卫**：用户一句话后，系统能披露至少一个 workforce 动作（接单/分支/模型/复核），证明"背后有 agent"而非空回答。

（其余守卫沿用 v4 §8。）

## 9. v1 → v5 范式演进（终局表）

| 版本 | 范式 | 致命缺陷（被下一版推翻）|
|---|---|---|
| v1 | 3 角色壳（仪表盘）| Phase1181 已砍侧边栏，问题不在这 |

**为什么 v5 是范式终点**：再往下不再是"设计决策"，而是"这个回话处的圆角/这个语气的措辞/这个 workforce 披露的动画"——这些**只有看真实渲染反复调才能定**，纸面迭代到此为止。

## 10. 诚实声明：这是范式终点，不是完美终点

v5 是**范式层面**的最终方案（对话协议是 chat-gateway 系统的必然归处）。但 v5 不等于"完美"：
- 具体的语气措辞、对话节奏、workforce 披露粒度，需在实施 P0-a 的真实渲染中调。
- 意图识别命中率仍依赖后端 LLM 路由（v3 §1.5 审计，现 regex 级，需升级）。
- 这些属"实施中迭代"，不再有 v6 范式方案。

**打破无限迭代循环的方式：v5 定稿 → 进入实施 → 在真实渲染中调优，而非继续写 v6。**

*v5 基于：①真实当前代码 `OwnerOSShell.js`（发现 Provider 术语外泄）；②真实截图 `phase1858-owner-os-home`（发现已是浅色）；③chat-gateway 系统本质（`/chat`、`/chat-gateway/execute`、WebSocket、`chatActionIntentParser`）；④顶级 agent 产品范式（无首页，只有第一句话）。这是第一次让方案"顺着系统的本质"，而非"强加一个外部范式"。*
