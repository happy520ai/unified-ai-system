# UI 重构最终方案：角色自适应壳（Role-Adaptive Shell）

> **日期**：2026-06-21
> **范围**：`unified-ai-system` 全部前端 UI（根目录静态 HTML + `apps/ai-gateway-service/src/ui/` SSR 组件库）
> **状态**：设计已确认方向，待 spec 审阅 → 实施计划
> **遵循契约**：`docs/design/codex-design-knowledge/PME_DESIGN.md`、`DO_NOT_DESIGN_LIKE_THIS.md`、`COMPONENT_PATTERNS.md`

---

## 0. 一句话总结

> 把"老板要的极简"和"运营要的全功能"从**同一个壳里的互相打架**，变成**三个角色壳的各取所需**——同一套后端、同一份数据，按"谁在用"呈现完全不同的界面，通过顶部全局开关切换，默认入口智能记忆。

---

## 1. 问题诊断（根因）

### 1.1 根本矛盾：两个互相打架的产品共用一个壳

文档库里存在**两套截然相反的设计哲学**，当前代码把两者硬塞进同一个侧边栏，这是"按钮墙"和"工程后台当首页"反复出现的根因。

| 维度 | 操作侧（`UI-DESIGN-PROPOSAL.md`）| 老板侧（`PME_DESIGN.md` / 设计知识包）|
|---|---|---|
| 定位 | "本地 AI 作战指挥室" | "本地老板 AI 总控台" |
| 目标用户 | 运营/工程师，每天操作 | 老板，偶尔看一眼 |
| 信息密度 | 高密度：三模执行器、30+ 模型、审批队列、证据流 | 极简：一个主按钮 + 三张结果卡 |
| 核心交付 | 实时状态、路由决策、证据可视化 | **中文日报**（"报告比页面更重要"）|
| 视觉 | 深色指挥舱、终端风、数据流 | 极简未来 OS、大留白、冷静 |
| 反模式 | 想要"作战室""大屏 cockpit" | 明文禁止"花哨 3D / 大屏 cockpit / 按钮墙" |

**结论：这不是一个 UI，而是两个产品共用了一个外壳。**

### 1.2 当前代码的具体症状

1. **导航扁平化**：`components/nav.js` 中 11 项 Core + 8 项 System = ~20 个顶级侧边栏项，无嵌套、无渐进披露 → 典型按钮墙。
2. **80 个孤立 SSR Panel**：`apps/ai-gateway-service/src/ui/components/` 下有 ~80 个 `*Panel.js`（OwnerBossViewPanel、MissionControlPanel、GodModePanel、ThreeModeOverviewPanel、WorkforceSchedulerPanel 等），是巨大的并行 UI 库存，但**基本没有接入实际服务的根目录页面**——是技术债/死 UI。
3. **三套色系并存**：
   - `console.css`：深紫玻璃拟态（`#7c6aef` 强调，`backdrop-filter: blur(60-80px)`）
   - `apps/.../ui/styles/ownerOsTheme.js`：深色青蓝（`#00e5ff` 青色 CTA）
   - `docs/design/pme-owner-ui-design.md`（文档）：却写的是浅色白底
   - → 文档与实现严重漂移。
4. **品牌名不一致**："太极北斗" / "Unified AI Gateway" / "NeuralForge Console" 混用。
5. **每页 CSS 重复**：`dashboard.html`、`index.html` 各自重新声明 `@keyframes fadeIn`、`.page-wrapper` 等，设计系统只部分集中，页面互相漂移。
6. **命令面板只在首页有效**，非全局。
7. **真实用户反馈（phase1881）**：滚动滚不动、找不到聊天框、首页有按钮但没有任务输入区、当前 Owner OS 老板无法直接使用。

### 1.3 文档自己反复警告的失败模式

- `DO_NOT_DESIGN_LIKE_THIS.md`：不做按钮墙、不做工程后台首页、不把模块平铺给 owner、不暴露 Phase/evidence/trace。
- `CODEX_UI_REWRITE_PLAYBOOK.md`：不要把设计稿/截图/自动化测试说成"真实可用"。
- `phase1939p`：UI 必须在"无人额外解释"的情况下可用——历史证明做不到。

## 2. 设计哲学：一个 IA，三张脸

### 2.1 核心主张——角色自适应壳（Role-Adaptive Shell）

同一套系统、同一套数据、同一套 API，但按"谁在用"呈现完全不同的界面。不再用扁平的 20 项侧边栏轰炸所有人，而是**按角色折叠信息**。

```
                    ┌─────────────────────────┐
                    │   同一个 AI Gateway 后端  │
                    │   同一份数据 / 同一套 API │
                    └────────────┬────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              ▼                  ▼                  ▼
      ╔═════════════╗   ╔═══════════════╗   ╔══════════════╗
      ║ 主人首页      ║   ║ 运营控制台      ║   ║ 高级模式抽屉   ║
      ║ Owner Home   ║   ║ Operator Console║   ║ Advanced Drawer║
      ╠═════════════╣   ╠═══════════════╣   ╠══════════════╣
      ║ 零学习        ║   ║ 作战指挥室      ║   ║ 默认折叠       ║
      ║ 一个主按钮    ║   ║ 三模/路由/审批   ║   ║ Phase/证据/trace║
      ║ 三张结果卡    ║   ║ 实时数据流      ║   ║ raw 路径       ║
      ║ 中文日报      ║   ║ 证据可视化      ║   ║               ║
      ╚═════════════╝   ╚═══════════════╝   ╚══════════════╝
           三个角色之一            运营角色内              任何页面底部
```

这正是 `PME_DESIGN.md` 里"三模分离"（owner / operator / engineering）的真正落地，也同时满足 `UI-DESIGN-PROPOSAL.md` 对功能完整覆盖的要求——只是**按角色路由，而不是平铺**。

### 2.2 三个已确认的决策（用户已选定）

| 决策点 | 选定方案 |
|---|---|
| 默认入口 | **智能判断 / 记忆**：记住上次角色选择，首次访问时根据信号智能推断（例如检测到本地桌面启动器来源 → 老板；检测到运营台 bookmark/深链 → 运营）|
| 角色切换 | **顶部全局开关**：顶部一个明显的 3 选 1 角色开关（老板 / 运营 / 工程），点击即刻切换整个外壳 |
| 功能范围 | **全量覆盖**：整合 80 个孤立 SSR Panel + 实现 UI-DESIGN-PROPOSAL 的全部 6 大功能区 |

## 3. 信息架构重构（IA）

### 3.1 从 20 个扁平项 → 3 个主入口 + 渐进披露

**当前（问题）**：
```
核心：chat | dashboard | routing | workforce | neurons | security | operations
     | apikeys | analytics | approval | tri-mode        ← 11 项平铺
系统：knowledge | governance | evidence | providerWizard | codexContext
     | team | settings | ecommerce                       ← 8 项平铺
```

**方案：按角色的三舱制**
```
┌─────────────────────────────────────────────────┐
│  [👁 老板视图]  [🎛 运营视图]  [⚙ 工程]   ← 顶部角色开关  │
├─────────────────────────────────────────────────┤
│                                                  │
│  ▌ 今日（所有角色可见，内容随角色变化）              │
│    主人首页（日报 + 一个主按钮 + 三张卡）           │
│                                                  │
│  ▌ 对话（所有角色可见）                            │
│    智能对话（三模：Normal/God/Tianshu）            │
│                                                  │
│  ▌ 指挥（仅运营/工程视图展开；老板视图折叠）         │
│    ├ 模型路由（30+ 模型 · 路由策略 · 成本）         │
│    ├ 员工劳动力（金字塔 · 协作 · 分支执行）          │
│    ├ 审批中心（队列 · 历史）                       │
│    └ 知识库（文档 · RAG 检索）                     │
│                                                  │
│  ▌ 安全（运营/工程可见）                           │
│    ├ 权限引擎（GVC）+ 审计日志                      │
│    └ 连接器（飞书/企微）                           │
│                                                  │
│  ▌ 设置                                          │
│    Provider 配置 · 密钥 · 备份                     │
│                                                  │
│  ▾ 高级模式（默认折叠抽屉，全局面）                  │
│    Phase · Evidence · Trace · Codex 上下文         │

```

**关键变化**：
- 老板视图下"指挥"整组折叠（不显示），运营/工程视图展开。
- 工程细节永远进底部抽屉，不作为顶级导航项。
- `apikeys` / `analytics` 合并进父级（设置 / 路由成本），不再是顶级项。
- **侧边栏从 20 项降到 5-6 项**。

## 4. 视觉系统统一（解决文档与实现的漂移）

### 4.1 统一为"冷静深空 + 单一强调色"，用密度区分角色

不再维护三套色系。统一为一个 `ownerOsTheme.css`，通过 `data-density="calm"|"dense"` 属性切换密度。

| Token | 老板首页（calm 密度）| 运营控制台（dense 密度）|
|---|---|---|
| 背景 | `#0a0e1a` 深空蓝，**大留白** | 同色，卡片更紧凑、网格更密 |
| 面板 | `rgba(255,255,255,0.025)` 玻璃 | 实线分隔（`#1e2438`），减少模糊以提升可读 |
| 强调色 | **单一**青色 CTA `#00e5ff`（PME 已定）| 同色 + 状态色 |
| 成功 | `#10b981` | 同 |
| 警告 | `#f59e0b` | 同 |
| 危险 | `#ef4444` | 同 |
| 文字主 | `--owner-ink`（接近白）| 同 |
| 文字辅 | `--owner-muted`（灰蓝）| 同 |
| 玻璃模糊 | `backdrop-filter: blur(60px) saturate(180%)` | `blur(20px)` 或无（密度优先）|
| 字体 | 中文优先：微软雅黑 / 苹方 / Segoe UI，**禁止远程字体** | 同 |
| 圆角 | 12px（PME 定）| 8px（密集场景）|
| 动效 | 淡入 / 计数，**无花哨 3D** | 状态点呼吸、数据流，克制 |

### 4.2 淘汰清单

- 删除 `console.css` 与 `ownerOsTheme.js` 中重复的色系定义，收敛到单一 token 源。
- 统一品牌名为 **"太极北斗"**（首页主标识）+ 副标 "Unified AI Gateway"，废弃 "NeuralForge Console" 混用。
- 删除 `consolePage.js` 的 ~17 个 `.bak-*` 备份。

## 7. 交互范式（三条铁律）

这套系统有"安全基因"（GVC 权限引擎、Phase 网关、审批流），交互模式必须与普通 SaaS 不同：

1. **审批驱动，而非点击即生效**——危险操作（provider 调用 / 文件变更 / 部署 / shell 命令）一律走"请求 → 预览 → 审批 → 执行"，UI 里危险按钮永远先弹预览。
2. **每个按钮必须给反馈**——`ButtonFeedbackState` 组件：点击后按钮旁持久显示 `已开始/已完成/已阻止/已跳过/失败`（中文）。消灭"死按钮"（Phase316A 存在的原因）。
3. **每个操作可追证据**——任何动作都有"查看证据"入口，展示安全断言（`providerCallsMade` / `secretValueExposed` / `chatModified` 等）。这是证据追踪能力的 UI 体现。

## 8. 实施路径（从现状到达）

### 8.1 总原则
- **不推倒重来**——保留 `console.css` 设计系统的优秀部分（token 化、深空玻璃拟态），只重构 IA、角色路由、收敛色系。
- **整合孤立的 80 个 Panel**——它们是对的功能、错的接入方式。把它们接到对应的角色壳里。
- **分阶段交付**，每阶段都有可用产出。

### 8.2 优先级

## 9. 验收标准（防止"设计稿说成真实可用"）

严格遵循 `CODEX_UI_REWRITE_PLAYBOOK.md` 的诚实原则：

- **按钮墙守卫**：任何首屏的顶级可点击元素 ≤ 3 个（主人首页）/ ≤ 8 个（运营台）。通过 `noButtonWallGuardPassed` 验证。
- **死按钮守卫**：每个 `<button>` 必须有 handler 且产生 `ButtonFeedbackState` 反馈。通过 Phase316A 验证。
- **老板零学习守卫**：主人首页首屏不含 Phase / verifier / evidence / trace / Provider Gate 字样。通过 `phase1939p` 无人解释可用性 gate。
- **角色隔离守卫**：老板视图下侧边栏 ≤ 3 项；工程细节只能在高级抽屉出现。
- **视觉漂移守卫**：全站只有一处 token 源；`console.css` 与 `ownerOsTheme.js` 无重复色系定义。
- **诚实声明**：不把本次重构的截图/自动化测试等同于"production-ready"或"老板已认可"。真实老板反馈走 `docs/owner-daily-use/records/` 手动台账，自动化不计入人工反馈。
