# UI 重构最终方案 v2：Agentic 任务流壳（深度渐进 + 空间劳动力）

> **日期**：2026-06-21（v2 全量重写核心架构）
> **范围**：`unified-ai-system` 全部前端 UI（根目录静态 HTML + `apps/ai-gateway-service/src/ui/` SSR 组件库）
> **状态**：v2 设计已确认方向，待 spec 审阅 → 实施计划
> **遵循契约**：`docs/design/codex-design-knowledge/PME_DESIGN.md`、`DO_NOT_DESIGN_LIKE_THIS.md`、`COMPONENT_PATTERNS.md`
> **相对 v1 的变化**：核心抽象从"3 个角色壳"推翻为"1 个壳 + 深度旋钮 + 任务执行流"；吸收 6 项升级（Agentic 范式、深度旋钮、模式带、日报推送、信任带、空间劳动力画布）。

---

## 0. 一句话总结

> 不再把 UI 当成"多页面仪表盘 + 角色开关"（2020 Grafana/Retool 范式）。
> 而是把它做成**一个 agent 系统该有的样子**：用户表达意图 → 实时看见 agent 在做什么 → 复核证据。同一套数据、同一个壳，用**深度旋钮**（精简/完整/深挖）连续渐进披露，不再按身份锁死。三模（Normal/God/Tianshu）从面板升格为**每条执行的 DNA**，证据从抽屉升格为**常驻信任带**，劳动力从表格升格为**空间活体画布**，日报从页面升格为**推送 + 桌面环境态**。

## 1. 问题诊断（根因，沿用 v1，仍成立）

### 1.1 根本矛盾：两个互相打架的产品共用一个壳

文档库里存在**两套截然相反的设计哲学**，当前代码把两者硬塞进同一个侧边栏，这是"按钮墙"和"工程后台当首页"反复出现的根因。

| 维度 | 操作侧（`UI-DESIGN-PROPOSAL.md`）| 老板侧（`PME_DESIGN.md` / 设计知识包）|
|---|---|---|
| 定位 | "本地 AI 作战指挥室" | "本地老板 AI 总控台" |
| 目标用户 | 运营/工程师，每天操作 | 老板，偶尔看一眼 |
| 信息密度 | 高密度：三模执行器、30+ 模型、审批队列、证据流 | 极简：一个主按钮 + 三张结果卡 |
| 核心交付 | 实时状态、路由决策、证据可视化 | **中文日报**（"报告比页面更重要"）|
| 视觉 | 深色指挥舱、终端风、数据流 | 极简未来 OS、大留白、冷静 |
| 反模式 | 想要"作战室""大屏 cockpit" | 明文禁止"花哨 3D / 大屏 cockpit / 按钮墙" |

### 1.2 v1 方案的自我批评（v2 推翻它）

v1 用"老板/运营/工程三个角色壳 + 顶部开关"解决了按钮墙。但它是**用 2020 的范式解决 2020 的问题**：
- 把一个 agent 系统做成"多页面仪表盘"，配不上它的独特性。
- 三模/劳动力/证据这些签名能力仍被当成"面板里的一项"，没被产品化。
- 角色是**身份锁**，但真实场景是老板有时想深挖、运营有时只要 10 秒摘要——把人钉死在一个角色里是限制。

### 1.3 当前代码的具体症状（沿用 v1）

1. **导航扁平化**：`components/nav.js` 中 11 项 Core + 8 项 System = ~20 个顶级侧边栏项 → 按钮墙。
2. **80 个孤立 SSR Panel**：`apps/ai-gateway-service/src/ui/components/` 下 ~80 个 `*Panel.js` 基本没接入实际页面。
3. **三套色系并存**：`console.css`（深紫玻璃）/ `ownerOsTheme.js`（深色青蓝）/ `pme-owner-ui-design.md`（文档写浅色）→ 严重漂移。
4. **品牌名不一致**：太极北斗 / Unified AI Gateway / NeuralForge Console 混用。
5. **每页 CSS 重复**、**命令面板只在首页有效**。
6. **真实反馈（phase1881）**：滚动滚不动、找不到聊天框、首页有按钮无任务输入区。

### 1.4 文档自己反复警告的失败模式

- `DO_NOT_DESIGN_LIKE_THIS.md`：不做按钮墙、不做工程后台首页、不暴露 Phase/evidence/trace。
- `CODEX_UI_REWRITE_PLAYBOOK.md`：不要把设计稿/截图/自动化测试说成"真实可用"。
- `phase1939p`：UI 必须在"无人额外解释"下可用。

## 2. 设计哲学（v2 核心，推翻 v1）

### 2.1 核心主张——Agentic 任务流壳（不是多页面仪表盘）

一个 agent 系统就该长得像 agent 系统。UI 的核心不再是"导航到某个页面填表执行"，而是**一条任务流**：

```
用户表达意图 → 实时看见 agent 在做什么 → 复核证据 → 完成
   (一句话)        (流式动作/谁接单/哪条分支)     (查看/介入)    (结果)
```

**这一个范式同时服务三类人**：
- 老板：只看流的"当前一步 + 结果"（深度=精简）
- 运营：看完整 agent 动作链（深度=完整）
- 工程：看每步证据 + 可展开 trace（深度=深挖）

→ 同一条流，按深度折叠/展开，**不按身份锁死**。

### 2.2 核心抽象——深度旋钮（取代 v1 的角色开关）

```
   ◐ 深度: [精简] ─●───── [完整] ────── [深挖]
```

| | 精简（老板日常）| 完整（运营常态）| 深挖（排障/工程）|
|---|---|---|---|
| 主区 | 三张结果卡 + 一个主 CTA | **任务执行流** + 实时指标 | 任务流 + 全量 trace |
| 信任带 | 只显总结句 | 显各项计数 | 显 raw phase/verifier |
| 任务流 | 只显当前一步 + 结果 | 显完整 agent 动作链 | 显每步证据 + 可展开 |
| 导航 | 隐藏"指挥"组 | 全展开 | 全展开 + 高级抽屉 |
| 劳动力 | 2D 坍缩态 | 空间画布默认态 | 空间画布 + 可点入节点 |
| 日报 | 顶部一张摘要卡 | 内嵌日报面板 | 完整日报 + 证据链接 |

**为什么比角色开关好**：
- 深度是**连续渐进**，角色是**离散跳变**。人是按当下需求调深度，不是按身份切换。
- 消除"我是老板所以看不见运营数据"的人为割裂。
- 默认值仍智能记忆（首次按来源推断：桌面启动器来源 → 精简；运营台深链 → 完整；之后记住）。
- 更现代（Notion AI / Linear 的 detail level 是这条路）。

**代价（诚实声明）**：比三按钮抽象，默认值若调不好会困惑"我在哪个深度"。缓解：首次访问有 5 秒引导 + 旋钮永远可见当前档位。

## 3. 信息架构（v2 重构）

### 3.1 一个壳，一个主任务流，顶部深度旋钮

```
╔══════════════════════════════════════════════════════════════╗
║  太极北斗 · 6月21日                       ◐深度[精简●─完整─深挖] ║
║  ── 信任带（常驻）────────────────────────────────────────     ║
║  ● 0真实Provider调用 · 0密钥暴露 · 47证据已封存 · 3员工在线     ║
╠══════════════════════════════════════════════════════════════╣
║                                                               ║
║  ┌─ 你的任务（主区，永远是焦点）────────────────────────┐     ║
║  │ ▎ "把昨天客户反馈归类，找前三类问题"                   │     ║
║  │                                                        │     ║
║  │ ─Tianshu─→ 判定中等复杂 → 走 God 多候选     [模式带]   │     ║
║  │ ▸ product-chief 接单 · 分支: 产品/工程/安全            │     ║
║  │ ▸ ▍读知识库…      ▍DeepSeek-V3 生成…    ◌待审批 1项    │     ║
║  │ [查看证据]  [介入]  [暂停]                              │     ║
║  └────────────────────────────────────────────────────────┘     ║
║                                                               ║
║  ┌─ 今天 ────────────────────────────────────────────┐        ║
║  │ ✅完成了什么    ⚠️发现了什么    ➜下一步点哪里         │        ║
║  └────────────────────────────────────────────────────┘        ║
║                                                               ║
║  ▌ 对话  ▌ 劳动力画布  ▌ 路由  ▌ 审批  ▌ 知识  ▌ 安全       ║
╚══════════════════════════════════════════════════════════════╝
```

**关键变化（对比 v1）**：
- ❌ 删除"顶部三个角色按钮" → ✅ 换成"一个深度旋钮"
- ✅ 新增**常驻信任带**（§6.5）
- ✅ **主区从"主人首页"换成"任务执行流"**（系统是 agent，主区就该是流，不是仪表盘）
- ✅ 三模变成**模式带**嵌在每条执行里（§6.3），不再是独立面板
- ✅ 日报变成**推送优先**（§6.4），UI 里只是回看

### 3.2 导航（侧边栏，按深度折叠）

```
▌ 今日（任务流 + 今天三卡，所有深度可见，内容随深度变化）
▌ 对话（三模对话，所有深度可见）
▌ 劳动力画布（精简→2D坍缩；完整/深挖→空间画布）
▌ 指挥组（精简深度下整组隐藏）
    ├ 模型路由（30+ 模型 · 路由策略 · 成本）
    ├ 审批中心（队列 · 历史）
    └ 知识库（文档 · RAG 检索）
▌ 安全（GVC 权限 · 审计 · 连接器）
▌ 设置（Provider · 密钥 · 备份）
▾ 高级模式（默认折叠抽屉，深挖深度下默认展开）
    Phase · Evidence · Trace · Codex 上下文
```

**侧边栏从 20 项 → 5-6 项**（指挥组在精简深度折叠）。

## 4. 视觉系统统一（沿用 v1，收敛三套色系）

### 4.1 统一为"冷静深空 + 单一强调色"，用密度切换

| Token | 精简密度 | 完整/深挖密度 |
|---|---|---|
| 背景 | `#0a0e1a` 深空蓝，大留白 | 同色，卡片更紧凑 |
| 面板 | `rgba(255,255,255,0.025)` 玻璃 | 实线分隔 `#1e2438`，减少模糊 |
| 强调色 | **单一**青色 CTA `#00e5ff` | 同 + 状态色 |
| 成功/警告/危险 | `#10b981` / `#f59e0b` / `#ef4444` | 同 |
| 玻璃模糊 | `blur(60px) saturate(180%)` | `blur(20px)` 或无 |
| 字体 | 中文优先：微软雅黑 / 苹方 / Segoe UI，**禁止远程字体** | 同 |
| 圆角 | 12px | 8px |

通过 `data-density="calm"|"dense"` 属性切换，单一 token 源（`ownerOsTheme.css` 收敛后）。

### 4.2 淘汰清单
- 删除 `console.css` 与 `ownerOsTheme.js` 重复色系定义。
- 统一品牌名 **"太极北斗"** + 副标 "Unified AI Gateway"，废弃 "NeuralForge Console"。
- 删除 `consolePage.js` 的 ~17 个 `.bak-*` 备份。

## 5. 六大升级（v2 核心，逐个落地）

### 5.1 升级 1：主范式 = 意图 → 执行流 → 复核（Agentic UI）

任务执行流是新主区，取代"主人首页当默认"。它直接消费后端已有能力：workforce-execution-fabric、三模、分支执行、证据链。

```
┌─ 你的任务 ──────────────────────────────────────────┐
│ "把昨天那批客户反馈归类，找出前三类问题"                │
│                                                      │
│ ─Tianshu─→ 判定: 中等复杂 → 走 God 多候选   [模式带]  │
│ ▸ product-chief 接单 · 分支: 产品/工程/安全           │
│ ▸ ▍正在读知识库…          ▍DeepSeek-V3 生成中…        │
│ ▸ ▍ux-researcher 复核…     ◌等待你审批 1 项            │
│                                                      │
│ [查看证据]  [介入]  [暂停]                            │
└──────────────────────────────────────────────────────┘
```

**深度旋钮如何作用于流**：
- 精简：只显"当前一步 + 结果"（老板看摘要）
- 完整：显完整 agent 动作链（运营监工）
- 深挖：每步可展开看 trace + 证据（工程排障）

**这是把后端的 agent 能力产品化**，而不是拆成三个面板。

### 5.2 升级 2：深度旋钮（取代角色开关，见 §2.2）

### 5.3 升级 3：三模从"面板"升格为"执行 DNA"（模式带）

每次执行都在动作旁带一条**模式带**，显示这件事是怎么被处理的：

```
翻译文档  ──Normal──→  DeepSeek 直出        ¥0.02
架构评审  ──God────→  3候选评审·无冲突       ¥0.18
客户分析  ──Tianshu─→  元路由判→God·复杂     ¥0.31
```

三模从"打开一个页面才能看到"变成**随处可见、解释每一次结果为什么是这个**。差异化能力产品化，而非功能堆叠。

### 5.4 升级 4：日报离开 UI（报告是推送，不是页面）

PME 说"报告比页面更重要"，v2 **字面意义上**当真：
- 8:00 自动生成中文日报 → 推送到企业微信 / 桌面通知 / 邮件。
- 老板**核心路径不开浏览器**（呼应 zero-learning 桌面启动器）。
- UI 里"日报"从主导航项 → 降级为"今天"区域内的回看入口。
- **环境态**：老板侧常驻桌面小组件/菜单栏图标，只显示"今天能用 / 待审批 N / 下一步"。

> 最好的 UI 是老板不用打开的 UI。

### 5.5 升级 5：证据从"高级抽屉"升格为"常驻信任带"

系统全部价值是信任与安全，把信任藏抽屉是反产品逻辑。加一条**永远可见的环境信任带**：

```
─────────────────────────────────────────────────────
  ● 今天 0 次真实 Provider 调用 · 0 密钥暴露 · 47 条证据已封存
─────────────────────────────────────────────────────
```

- 一眼建立信任，不需点开任何东西。
- 详细 trace 仍在高级区，但**信任感默认在线**。
- 这是证据链能力的"门面"。

### 5.6 升级 6：劳动力 = 空间活体画布（带 PME 红线边界）

⚠️ **与 PME "不做花哨 3D / 大屏 cockpit" 的红线冲突，必须定边界。**

**PME 红线真实意图是禁止"装饰性堆砌"**（为大屏而大屏、3D 为炫技）。它不是禁止"用空间表达真实结构"。劳动力系统**本身就是空间结构**（7 层金字塔 + 三路分支合并），用空间画布表达它是**信息可视化**，不是 cockpit 装饰。

**给画布定三条约束，让它合法过红线**：
1. **可降维**——画布一键坍缩成 2D 组织图（深度=精简时自动坍缩，守红线）。
2. **零装饰元素**——每个节点/连线/动画都映射真实员工状态/消息/分支。头像呼吸=在线、连线流动=有消息、分叉=真实分支执行。无纯装饰。
3. **默认静态、交互才活**——不持续渲染消耗注意力，hover/飞入才动。

```
        系统治理者 (L0)
       /          \
   产品主管      安全主管        ← L2 分支点
      |             |
  产品工程师    安全审计员        ← L3
      |
  ┌─ 任务: 客户反馈归类 ────────┐
  │   ╱       ╲        ╲       │  ← 分支执行（交互才动画）
  │ 产品     工程      安全     │
  │  ●        ●        ◌       │  ← ●在线 ◌待命
  └────────────────────────────┘
        ↓ 合并 ↓
      最终建议
```

> 一句话：**这是"用空间说真话"，不是"用空间炫技"。** 验收时卡这条边界。

## 6. 组件模式（沿用 PME 知识包 + 新增 Agentic 组件）

### 6.1 沿用（PME 已定义，不改动）
`OwnerPrimaryAction`、`OwnerStatusCard`（三张卡）、`OwnerDailyReport`、`NextActionCard`、`AdvancedModeDrawer`、`FailureFriendlyMessage`、`ButtonFeedbackState`。

### 6.2 新增（Agentic 范式所需）
- `DepthDial` — 顶部深度旋钮（精简/完整/深挖），状态持久化 localStorage，首次智能推断。
- `TrustRibbon` — 常驻信任带（环境态安全断言）。
- `TaskExecutionStream` — 主区任务执行流（消费 workforce-execution-fabric + 三模 + 证据）。
- `ModeRibbon` — 模式带（每条执行旁显示 Normal/God/Tianshu 处理方式）。
- `ApprovalInlineAction` — 内联审批（执行流中遇到审批直接就地批准/拒绝，不必跳页）。
- `WorkforceSpaceCanvas` — 空间劳动力画布（带三条 PME 边界约束）。
- `WorkforceOrgChart` — 画布的 2D 坍缩态（精简深度用）。
- `DailyReportPushChannel` — 日报推送通道（企微/桌面/邮件）+ UI 回看入口。
- `DesktopAmbientWidget` — 老板侧桌面环境态小组件。
- `EvidenceTimeline` — 证据按 Phase 时间线（住高级抽屉）。
- `RoutingPreviewPane` — 输入任务 → 实时预览路由决策。
- `ModelDirectoryGrid` — 30+ 模型按供应商分组目录。
- `BranchExecutionGraph` — 分支执行（产品/工程/安全 → 合并）流程图。

### 6.3 孤立 Panel 整合映射

| 孤立 Panel | 接入到 |
|---|---|
| OwnerOSShell / OwnerBossViewPanel / OwnerHeroCommand / OwnerNeuralSkillPreviewPanel | 任务流主区（精简深度下的摘要卡）|
| MissionControlPanel / ThreeModeOverviewPanel / GodModePanel / TianshuModePanel | TaskExecutionStream + ModeRibbon |
| ModelRoutingPanel | RoutingPreviewPane + ModelDirectoryGrid |
| WorkforceSchedulerPanel | WorkforceSpaceCanvas |
| （其余审批/证据/知识 Panel） | 内联审批 / 证据时间线 / 知识库 |

⚠️ 注意：TaskExecutionStream 是**新主区**，不是把某个 Panel 挪过去；许多 Panel 的内容要拆散重组进流里。这是整合难点。

## 7. 交互范式（三条铁律，沿用 v1）

1. **审批驱动，而非点击即生效**——危险操作走"请求 → 预览 → 审批 → 执行"。
2. **每个按钮必须给反馈**——`ButtonFeedbackState`（已开始/已完成/已阻止/已跳过/失败，中文）。
3. **每个操作可追证据**——任何动作有"查看证据"入口，展示安全断言。

## 8. 实施路径（v2 更重，分 3 个迭代）

⚠️ **诚实声明**：v2 工作量远超 v1。任务流 + 空间画布 + 推送管道 + 桌面小组件，P0 就会重。需要分 2-3 个迭代。

| 迭代 | 工作项 | 价值 |

## 9. 验收标准（防止"设计稿说成真实可用"）

严格遵循 `CODEX_UI_REWRITE_PLAYBOOK.md` 的诚实原则：

- **按钮墙守卫**：任何首屏顶级可点击 ≤ 3（精简）/ ≤ 8（完整）。
- **死按钮守卫**：每个 `<button>` 有 handler + `ButtonFeedbackState` 反馈（Phase316A）。
- **老板零学习守卫**：精简深度首屏无 Phase/verifier/evidence/trace/Provider Gate 字样（phase1939p）。
- **深度渐进守卫**：深度旋钮切换不丢失上下文（同一任务在三档深度下是渐进展开，不是三个页面）。
- **信任带常驻守卫**：信任带在任何深度都可见，至少显示总结句。
- **空间画布红线守卫**：①可坍缩成 2D ②零装饰元素（每视觉元素映射真实状态）③默认静态。不满足则判为 cockpit 装饰，驳回。
- **视觉漂移守卫**：全站只有一处 token 源。
- **诚实声明**：不把本次截图/自动化测试等同于"production-ready"或"老板已认可"。真实老板反馈走 `docs/owner-daily-use/records/` 手动台账，自动化不计入人工反馈。

## 10. v1 → v2 决策变更记录（便于追溯）

| 决策点 | v1（已废弃）| **v2（当前）** | 变更理由 |
|---|---|---|---|
| 核心抽象 | 3 个角色壳 + 顶部开关 | **1 个壳 + 深度旋钮** | 角色是身份锁，深度是连续渐进；更诚实更现代 |
| 默认主区 | 主人首页 | **任务执行流** | agent 系统主区就该是流，不是仪表盘 |
| 三模 | 一个面板 | **模式带，嵌在每条执行** | 差异化能力产品化 |
| 证据 | 藏高级抽屉 | **常驻信任带** | 信任是核心价值，该默认在线 |
| 日报 | 一个页面 | **推送优先 + 桌面环境态** | "报告比页面重要"字面当真 |
| 劳动力 | 表格 | **空间活体画布（可坍缩）** | 签名概念配得上空间表达，带红线约束 |
| 交互范式 | 导航+点击 | **意图→流→复核（Agentic）** | 越过 2020 仪表盘范式 |
| 所处世代 | 2020 Grafana/Retool | **2025-2026 Agentic/Ambient** | — |
